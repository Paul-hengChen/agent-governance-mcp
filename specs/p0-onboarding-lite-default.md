# Spec: P0 Onboarding (`agc init`) + Default to Lite Mode

> @pm · 2026-05-20
> Source: `research/value-proposition.md` §Recommendation P0 + Open Question #1

## Problem Statement

New users face a 6-step onboarding (understand MCP → configure IDE → create `.current/` → format `tasks.md` → learn 10 tools / 7 roles → grok constitution) before getting any value. Meanwhile, `/teamwork` (full multi-role chain) is the default entry, but solo-dev usage dominates and the multi-role chain is overkill for daily work — costing ~5× tokens vs `/teamwork-lite`. Two cuts reduce friction: (1) a `agc init` CLI that scaffolds the workspace files in one command, and (2) flipping the SessionStart hook default to inject the lite coordinator skill instead of the full one.

## User Stories

- As a new solo developer, I want to run `npx agent-governance-mcp init` (or `agc init`) in a fresh project, so that `.current/handoff.md`, `tasks.md`, and `.current/.config.json` exist with sane defaults without me copy-pasting templates.
- As a returning user, I want `agc init` to refuse to overwrite my existing handoff/task files, so that I don't accidentally lose state.
- As a solo developer opening a managed workspace in Claude Code, I want the SessionStart hook to inject the lite coordinator skill by default, so that my session boots with the smaller prompt and direct-execute SOP.
- As a team developer who needs the full chain, I want to opt back into full coordinator mode via an env var (`AGC_DEFAULT_SKILL=full`), so that the lite default doesn't block multi-role workflows.

## Acceptance Criteria

- **AC1 — `agc init` happy path**
  - Given an empty workspace (no `.current/`, no `tasks.md`)
  - When the user runs `agc init` (or `node bin/agc-init.mjs`) in that directory
  - Then `.current/handoff.md`, `.current/.config.json`, and `tasks.md` are created with the templates documented below, and the command exits 0 with a one-line success message naming the files created.

- **AC2 — `agc init` is idempotent / non-destructive**
  - Given a workspace where any of `.current/handoff.md`, `.current/.config.json`, or `tasks.md` already exists
  - When the user runs `agc init`
  - Then the existing file is left untouched (byte-for-byte unchanged), the command prints which files were skipped, and exits 0. No `--force` flag in v1.

- **AC3 — `agc init` writes valid initial state**
  - Given a fresh scaffold from AC1
  - When `tw_get_state` is called against the workspace
  - Then it parses successfully and returns `status: "Not_Started"`, `last_agent: "pm"`, empty `completed_tasks`, empty `pending_notes`, `qa_round: 0`.

- **AC4 — CLI is exposed via npm bin**
  - Given the repo installed via `npx github:Paul-hengChen/agent-governance-mcp` or `npm install`
  - When the user invokes `npx agc init`
  - Then the init script runs (i.e. `package.json` `bin` entry maps `agc` → `bin/agc-init.mjs`), with no requirement to know the file path.

- **AC5 — SessionStart hook defaults to lite skill**
  - Given a managed workspace and no `AGC_DEFAULT_SKILL` env var
  - When the SessionStart hook fires
  - Then `additionalContext` contains the contents of `content/skill-coordinator-lite.md` (not `skill-coordinator.md`) and the introductory prose says "Coordinator-Lite mode" with guidance to escalate via `/teamwork` for cross-module work.

- **AC6 — Full mode opt-in via env var**
  - Given `AGC_DEFAULT_SKILL=full` is set in the Claude Code session env
  - When the SessionStart hook fires
  - Then `additionalContext` contains `skill-coordinator.md` (full coordinator) and the prose matches the current full-mode behavior.

## Out of Scope

- Writing or modifying `~/.claude/settings.json` or any IDE MCP config (Init scope = file scaffold only; user wires MCP themselves).
- Sample constitution generation — the scaffold points at the server-provided `content/constitution.md` via the existing hook lookup; no per-workspace copy is created.
- `--force` / `--overwrite` flags on `agc init`. Re-init must be a manual delete + re-run.
- Web dashboard, RBAC, npm publishing (those are P1/P2 in the research report).
- Renaming the `teamwork` prompt id or marking it deprecated — both prompts (`teamwork`, `teamwork-lite`) keep their current behavior; only the hook default changes.

## Dependencies / Prerequisites

- None blocking. Existing `bin/agent-governance-context.mjs` is the only file that needs hook-default changes. New file `bin/agc-init.mjs` is additive.
- v3.6.x lite skill (`content/skill-coordinator-lite.md`) already exists.

## Templates (for AC1 / AC3)

`.current/handoff.md`:
```
---
schema_version: 1
active_feature: ""
status: "Not_Started"
last_updated: "<ISO timestamp at init>"
last_agent: "pm"
qa_round: 0
---
# 📍 任務交接狀態 (Handoff State)

## ✅ 已完成 (Completed)

## ⚠️ 待辦與交接 (Pending & Handoff Notes)

---
> 🤖 **System Note**: Auto-generated by agent-governance-mcp. Do NOT edit manually.
```

`.current/.config.json`: defer to existing default (whatever `tools/config.ts` produces when the file is absent). Init writes the literal `{ "schema_version": 1 }` so first read doesn't trigger a migration warning.

`tasks.md`:
```
# Tasks

<!-- Append via tw_add_task. -->

## Completed

<!-- tw_complete_task will move items here -->
```
