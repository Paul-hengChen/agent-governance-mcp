# a11-escalation-grammar

## Problem Statement
Every skill file re-spells its own `tw_update_state(status=Blocked, pending_notes=["<Role> blocked: <reason>", "next_role: <role>"])` incantation inline, with slightly different phrasing per site, and normative gate rules are written as free-form prose rather than a scannable trigger → action → escape shape. `docs/backlog.md` A11 estimated this at "~40% of `skill-architect.md`" boilerplate; a fresh grep-and-read survey of the current (post-A6/A7/A13) files found the real count is far smaller — 5 escalation sites in a 77-line `skill-architect.md` (~9%), and a total of roughly 20 scattered sites across 7 files (`skill-architect.md`, `skill-sr-engineer.md`, `skill-qa-engineer.md`, `skill-design-auditor.md`, `skill-code-reviewer.md`, `skill-coordinator.md`, `skill-release-engineer.md`). `skill-qa-visual.md` (A6) and `skill-pm.md` (A7) were already consolidated into table form by prior tickets and are the working exemplars this ticket generalizes, not new work. `skill-doc-writer.md` and `skill-researcher.md` carry only a single closing `next_role:` line each and are out of scope — a one-row table would add overhead, not remove it.

## User Stories
- As an agent following a skill SOP, I want every "if X, block/fail with Y, route to Z" rule expressed as one table row, so I can scan the escalation surface without re-parsing numbered-step prose.
- As a maintainer changing the escalation call shape (e.g. adding a new note-token convention), I want the call format defined once in the constitution, so I edit one bullet instead of grepping every skill file.
- As a downstream parser/reader (human or agent) of `pending_notes`, I want a consistent `status` / note-token / `next_role` shape across skills, so near-identical escalations don't carry different phrasing that looks like drift.

## Acceptance Criteria
- **AC1** — Given `content/const-05-core-standards.md` §3 State Synchronisation, when the file is read, then it contains exactly one canonical **Escalation call format** bullet specifying the shape `tw_update_state(status=<Blocked|FAIL>, agent_id=<role>, pending_notes=["<Role>: <situation> — <detail>", "next_role: <role>"])` and cross-references the existing note-token vocabulary already in use elsewhere (`visual_fail:`, `review: APPROVED`/`CHANGES_REQUESTED`, `resume_of:`, `scope_decision_why`, `covers:`) without redefining them.
- **AC2** — Given each of the 7 target skill files (`skill-architect.md`, `skill-sr-engineer.md`, `skill-qa-engineer.md`, `skill-design-auditor.md`, `skill-code-reviewer.md`, `skill-coordinator.md`, `skill-release-engineer.md`), when its escalation sites are enumerated, then every one appears as exactly one row in an `## Escalation Routes` table with columns `| situation | status | note token | next_role |`, and the surrounding SOP prose references the table by situation name rather than re-stating the full `tw_update_state(...)` call.
- **AC3** — Given `skill-pm.md` (A7 Gate Summary table) and `skill-qa-visual.md` (A6 error-code/STOP-route table), when reviewed post-edit, then their existing table shapes and column headers are byte-unchanged; only a cross-reference line pointing at the new constitution bullet is added.
- **AC4** — Given the full edit set, when `npm run build && npm test` runs, then every previously-green suite stays green, including `test/compose-equivalence.test.mjs` (golden fixtures regenerated if `const-*.md` bytes changed), `test/context-budget.test.mjs` (hardcoded `~tok` caps re-baselined with a documented old→new comment ONLY if tripped — qa-owned bump per the `C2-06` precedent), `test/constitution-deliverable-guard.test.mjs`, `test/error-code-contract.test.mjs`, `test/skill-evolution-v3.11.test.mjs`, and `test/widget-shape-spec.test.mjs`.
- **AC5** — Given `test/error-code-contract.test.mjs`'s doc-side harvest (every gate-error-code-shaped backtick token in `content/*.md` must be a documented, registry-recognized code), when the escalation-site restructuring lands, then no error-code token present today (e.g. `VISUAL_EVIDENCE_MISSING`, `MISSING_REVIEW_EVIDENCE`, `AGENT_ID_REQUIRED`, `QA_ROUND_EXCEEDED`, `REVIEW_ROUND_EXCEEDED`, `BASELINE_MANIFEST_MISSING`, `BASELINE_PROVENANCE_INCOMPLETE`) is removed from `content/*.md` — each must still appear verbatim, whether inside a table cell or surrounding prose.
- **AC6** — Given a normative gate rule being touched by this ticket (i.e. one of the enumerated escalation sites), when rewritten, then its trigger/action/escape reads as WHEN `<trigger>` → DO `<action>` → ELSE `<escape>` — either as literal prose in that shape, or as one row of the `## Escalation Routes` table (table row = the WHEN/DO/ELSE shape collapsed to columns). Gate rules NOT touched by this ticket (e.g. `skill-pm.md`'s Gate Summary, already A7-shaped; unrelated non-escalation prose) are out of scope for reshaping.
- **AC7** — Given `docs/backlog.md`'s A11 row, when this ticket ships, then it is marked done, citing the corrected incantation-count finding (the "~40%" estimate was stale relative to post-A6/A7/A13 files) and the mechanism (one constitution bullet + 7 per-file Escalation Routes tables + 2 light-touch cross-reference edits).

## Out of Scope
- `skill-doc-writer.md` and `skill-researcher.md` — each carries a single closing `next_role:` line, no scattered escalation incantations; a table would add overhead, not remove it.
- `skill-coordinator-lite.md` — zero `status=Blocked`/`next_role` hits (server-read-only mode, no state writes); nothing to consolidate.
- Full WHEN/DO/ELSE rewrite of every normative rule constitution-wide (e.g. the Visual Harness Gate prose in `skill-architect.md`'s Artifact Schema, the Gate Summary rows in `skill-pm.md` already shaped by A7). This ticket reshapes the **escalation call sites** enumerated in AC2 plus the constitution's one canonical definition — not a constitution-wide prose pass. A wider WHEN/DO/ELSE style migration, if wanted later, is a separate ticket.
- Any change to `tools/transitions.ts`, `gates/registry.ts`, or other TypeScript sources — this is a content-only ticket (markdown in `content/` + `docs/backlog.md`).
- Renumbering or restructuring SOP step sequences beyond what's needed to reference the new table (e.g. `skill-architect.md`'s steps stay 1-7, they just point at table rows instead of inlining the call).
