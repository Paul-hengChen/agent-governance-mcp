# v3.15.0 — Server-enforced widget verification + writeState refactor + sentinel symmetric fix

## Problem Statement

Two intentional deferrals from v3.14.0 and one v3.14.1 patch-trio observation now come due. (1) **R6 server enforcement** — v3.14.0 architecture §A intentionally downgraded widget-shape verification to SOP-level only, reserving `VISUAL_WIDGETS_UNVERIFIED` error code for v3.15.0; operator dishonesty (writing `[x]` falsely in `qa_reports/visual_<id>.md`) cannot be detected by current server. (2) **`writeHandoffState` positional-param growth** — 11 positional parameters across the storage interface (post-v3.14.0); each new round counter or feature field worsens call-site readability and increases edit-site risk; code-reviewer flagged for v3.15.0 in `review_reports/review_T105-T107.md`. (3) **`qa_round` / `review_round` Round 4 sentinel symmetric fix** — v3.14.1 fixed the `visual_round` Round 6 sentinel from `=== && ===` to `>= && <` cap-cross predicate; the same off-by-one exists at `index.ts:747-752` for `qa_round` / `review_round` (path-to-trigger is migration-only, NOT blocking, but symmetric is honest). Locked Question Batch decisions from session opening v3.15.0: (a) **parse markdown checkbox** for R6 (server reads visual_<id>.md, greps `^-\s+\[\s\]`); (b) **dual API** for writeState (positional `@deprecated`, options-object new); (c) v3.14.2 sentinel fix **merges into v3.15.0** (no standalone patch release).

## User Stories

- As a **framework operator**, I want the server to verify Widget Shape Checklist completeness in `qa_reports/visual_<task-id>.md` so that PASS cannot be issued when the qa-engineer (or a buggy automation) wrote unchecked `[ ]` rows — closing the last operator-dishonesty hole that v3.14.0 architecture §A explicitly deferred.
- As a **future maintainer**, I want `writeHandoffState` and `HandoffStorage.writeState` to accept an options object so that adding a new field (next round counter, new evidence type, etc.) doesn't require touching every call site and risk parameter-order mistakes.
- As a **caller of the existing 11-positional API**, I want the positional signature to keep working with an `@deprecated` warning so that my code migrates on my schedule, not on v3.15.0's.
- As a **framework maintainer**, I want the `qa_round` / `review_round` Round 4 sentinels to use the same `>= && <` cap-cross predicate as v3.14.1's `visual_round` Round 6 sentinel so that all three counters behave identically and a future external-bump (migration / hand-edit) can't skip the lock notice.

## Acceptance Criteria

### R6 — server-enforced widget verification

- **AC-1**: `Given` `design/<active_feature>.md` declares `## Visual Baselines` AND `qa_reports/visual_<task-id>.md` exists for every completed_task, `When` the visual report contains a `## Widget Shape Verification` H2 with at least one row `- [ ] <widget-id>` (unchecked), `Then` the server MUST reject the PASS write with `{ error: "VISUAL_WIDGETS_UNVERIFIED", attempted, allowed, hint: "Unchecked widget row(s) in qa_reports/visual_<id>.md: <list>" }`.
- **AC-2**: `Given` the same gate-active state, `When` the visual report contains ONLY `- [x] …` rows (all checked) OR contains no `## Widget Shape Verification` H2 at all (backwards-compat: pre-v3.15.0 reports without the checklist section), `Then` the server MUST accept the PASS (gate passes through).
- **AC-3**: `Given` a v3.14.0-style visual report (no `## Widget Shape Verification` H2 — the section is optional per skill-qa-visual SOP), `When` PASS attempts, `Then` AC-2 backwards-compat applies. **No retroactive flagging**.
- **AC-4**: `Given` the gate fires, `Then` the error response MUST list every offending task-id + every unchecked widget-id inline so the operator can fix in one round-trip (not require multiple PASS attempts to discover the next missing checkbox).
- **AC-5**: Parsing MUST be permissive on whitespace and case-sensitive on bracket content — `- [ ]`, `- [x]`, `- [X]` are recognised; `- [Y]` is treated as unchecked (an operator typo that the gate catches rather than silently accepting).

### writeState options-object refactor (dual API)

- **AC-6**: `tools/handoff.ts` MUST export an additional overload `writeHandoffState(opts: WriteHandoffStateOptions): Promise<string>` where `WriteHandoffStateOptions` is an object containing every current positional parameter as a named field. The existing 11-positional signature MUST remain functional with no behaviour change.
- **AC-7**: `tools/storage.ts` `HandoffStorage` interface MUST gain the options-object signature alongside the positional one (TypeScript overloads or a discriminated parameter). `FileHandoffStorage` and `SqliteHandoffStorage` implementations both support both signatures.
- **AC-8**: `index.ts` handler MUST switch its `storage.writeState(...)` call to the options-object form as the migration reference point. Other callers (tests etc.) MAY continue using positional; positional callers receive a TSDoc `@deprecated` annotation with migration guidance.
- **AC-9**: `@deprecated` JSDoc MUST appear on the positional signature with: (a) "v3.15.0: prefer the options-object overload", (b) migration example, (c) "positional signature retained for backwards-compat; planned removal in v4.0.0".
- **AC-10**: Backwards-compat: `Given` a caller passes 8 positional args (the pre-v3.9.0 baseline), `When` the function runs, `Then` `qaRound` / `prdPath` / `reviewRound` / `visualRound` MUST default to their existing defaults (0 / undefined / 0 / 0). The dual API MUST NOT regress this.

### Round 4 sentinel symmetric fix

- **AC-11**: `index.ts:747` `qa_round` sentinel predicate changes from `new_qa_round === 4 && prev_qa_round === 3` to `new_qa_round >= 4 && prev_qa_round < 4` — symmetric to v3.14.1's `visual_round` Round 6 fix.
- **AC-12**: `index.ts:750` `review_round` sentinel predicate changes from `new_review_round === 4 && prev_review_round === 3` to `new_review_round >= 4 && prev_review_round < 4` — same predicate fix.
- **AC-13**: Both sentinel messages are unchanged (Round 4 / Review Round 4 wording preserved). Only the predicate changes.

- **AC-15**: `npm audit --audit-level=high` carries the v3.14.1 reframed waiver unchanged (the `embedding_model` allowlist + path-closed framing).
