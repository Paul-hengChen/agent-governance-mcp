> Spec version: 1.0.0
> Date: 2026-06-09
> Author: pm subagent (claude-sonnet-4-6)
> Status: In_Progress → next_role: architect

---

## Problem Statement

`agc init` currently scaffolds only the state-tracking artifacts (`.current/handoff.md`,
`.current/.config.json`, `tasks.md`). It does nothing to tell Claude Code, Codex, or
Antigravity that the governance server exists or how to engage with it. The result is N
hand-maintained per-agent files — `CLAUDE.md`, `AGENTS.md`, `.antigravityrules` — each
containing duplicated governance rules that drift independently from `content/constitution.md`,
the single source of truth. This is the exact doc-vs-code drift class that Constitution
v3.27.0 was created to close; it now manifests at the governance layer itself. The fix is to
extend `agc init` to generate thin per-agent ENTRY adapters (pointers + execution profiles,
never copies of the rules) and to add `agc check` so that any deployed adapter can be
detected as stale relative to the installed server version.

---

## User Stories

1. As a developer setting up a new managed workspace, I want `agc init` to generate
   `AGENTS.md`, `.antigravityrules`, and a `CLAUDE.md` entry section automatically, so that
   every agent that opens the project immediately knows to engage the governance server without
   me hand-authoring per-agent files.

2. As a developer who ran `agc init` on a workspace before this feature shipped, I want
   re-running `agc init` to skip adapter files that already exist (idempotent), so that
   customised content is never clobbered.

3. As a developer on a project that has been open for several weeks, I want `agc check` to
   warn me when a generated adapter file is stamped with an older server version than the one
   currently installed, so that stale adapter content is never silent.

4. As Codex running in a managed workspace, I want `AGENTS.md` to contain a clear, concise
   loader instruction and an execution profile for Codex, so that I know how to start a
   governed session without reading any other file.

5. As Antigravity running in a managed workspace, I want `.antigravityrules` to contain the
   same loader instruction and an Antigravity-appropriate execution profile, so that governance
   coverage is not Claude-only.

6. As Claude Code running in a managed workspace, I want the generated `CLAUDE.md` entry
   section to be a pointer to the governance server, not a second copy of the constitution,
   so that the CLAUDE.md file remains concise and the constitution stays authoritative.

## Acceptance Criteria

### AC-1 — Template files exist in the repository

Given the repository is at any commit after this feature ships,
When a developer browses `templates/agent-adapters/`,
Then they find exactly three files: `claude.md`, `codex.md`, `antigravity.md`,
each containing a loader section and an execution-profile section, with a version-stamp
placeholder `{{AGC_VERSION}}`, and NO copied constitution clauses.

### AC-2 — `agc init` writes all three adapters

Given a workspace directory that has none of `CLAUDE.md`, `AGENTS.md`, `.antigravityrules`,
When the developer runs `agc init` in that directory,
Then all three adapter files are created, the `{{AGC_VERSION}}` placeholder is replaced
with the installed server version (from `package.json`), and stdout reports them as created.

### AC-3 — `agc init` is idempotent for adapter files

Given a workspace where `AGENTS.md` already exists with custom content,
When the developer runs `agc init`,
Then `AGENTS.md` is unchanged and stdout reports it as skipped, while the other two adapter
files are created normally (same behavior as existing file-skip logic for `handoff.md` etc.).

### AC-4 — Version stamp is present and readable

Given an adapter file generated by `agc init`,
When the file is opened in any text editor,
Then the first non-blank, non-comment line or a clearly labelled header line contains
`agc-version: <semver>` where `<semver>` matches the installed `package.json` version at the
time `agc init` was run.

### AC-5 — `agc check` detects stale adapters

Given a workspace with adapter files stamped with version `X`,
And the installed `agent-governance-mcp` version is `Y` where `Y !== X`,
When the developer runs `agc check` in that workspace,
Then the command prints a warning identifying each stale adapter file and the version gap,
and exits with code 1.

### AC-6 — `agc check` exits 0 when all adapters are current

Given a workspace with adapter files all stamped with the currently installed version,
When the developer runs `agc check`,
Then stdout reports all adapters are current and the command exits with code 0.

### AC-7 — `agc check` exits 0 and is silent when no adapters are present

Given a workspace that has no adapter files (e.g. pre-init),
When the developer runs `agc check`,
Then the command exits with code 0 and produces no output (no false alarms on unmanaged
projects).

### AC-8 — Loader text is a pointer, never a rule copy

Given any generated adapter file for any agent,
When its content is diffed against `content/constitution.md`,
Then zero lines from the constitution body appear verbatim in the adapter (the loader is
pointer-only; execution-profile lines about this agent's capabilities are permitted).

### AC-9 — `agc init` sub-command routing is extended

Given the user runs `agc check`,
When the `agc-init.mjs` dispatcher reads `process.argv[2]`,
Then it routes to the check handler (same binary, new sub-command), and the usage string is
updated to list both `init` and `check`.

## Out of Scope (Deferred)

The following are explicitly excluded from this MVP. Each is a named follow-up.

| item | rationale for deferral |
|---|---|
| `agc update` — auto-refresh stale adapters in-place | MVP only needs to detect staleness (agc check exits 1). Auto-refresh requires a merge strategy when the user has edited the file after init; that decision needs a separate PM cycle. |
| Mode A — live-reference (runtime fetch) vs Mode B — stamped copy negotiation | MVP always uses Mode B (stamped copy + agc check). Mode A requires per-agent verification that the agent can load an MCP prompt into context automatically; Antigravity and Codex are unverified on this point (T3 confidence per research). |
| Auto-detecting which agents a project uses (`--agent` flag or marker detection) | MVP writes all three adapter files unconditionally (idempotent skip if present). Detection logic adds complexity with low payoff at MVP; the three files are small and non-destructive. |
| Global config management (`~/.codex/AGENTS.md`, Antigravity global) | Per the 2026-06-09 Codex review convergence: global config holds personal preferences only. agc init must NOT write global configs. Out of scope permanently for this feature. |
| Antigravity-specific capability verification (subagent dispatch, prompt auto-load) | Research report cites T3 confidence for Antigravity. The adapter is generated with a "unverified — fall back to tw_switch_role" note; full verification is a follow-up research task. |
| Watermark relocation (moving §1 Watermark subagent-detection from constitution to claude adapter) | Identified as a worthwhile cleanup in the research report but with non-trivial blast radius (touches skill-coordinator.md + watermark validation util). Separate feature cycle. |
| Cursor support (`.cursorrules`) | Not in the three-party consensus scope. Add in a subsequent adapter-expansion feature. |
| `agc init` writing adapters to a subdirectory (`--root` flag) | Not needed at MVP; cwd-based behavior matches the existing init pattern. |
| Constitution pruning (removing agent-specific mechanics from the universal contract) | A meaningful improvement, but orthogonal to adapter scaffolding. Separate feature. |

## Dependencies / Prerequisites

1. **`package.json` is the version source for the stamp.** The version stamp written into each
   adapter is read from `package.json` at `agc init` runtime (same source as
   `scripts/check-version.mjs`). No new version file is introduced.

2. **`agc check` version comparison logic.** `agc check` reads the `agc-version:` comment from
   each adapter file and compares it to the installed package version. The installed version
   is resolved by walking `require.resolve("agent-governance-mcp")` or by reading
   `package.json` from the directory containing the `agc` binary. Architect must specify the
   exact resolution path (works for both local dev and `npx github:...` installs). This is a
   design decision for the architect phase.

3. **External references (Resource Audit):**
   - `https://developers.openai.com/codex/guides/agents-md` — classified `index`. The research
     report already extracted the load-bearing facts (AGENTS.md is auto-loaded, global+project
     layering supported). Re-fetch not required for spec or implementation.
   - `https://developers.openai.com/codex/hooks` — classified `index`. Referenced only to
     establish that hooks are a separate lifecycle layer; no implementation dependency.

4. **`agc-init.mjs` sub-command routing.** The current binary uses a single positional arg
   `process.argv[2] === "init"`. Extending to `check` requires a dispatcher. The architect
   should decide: expand `agc-init.mjs` in-place, or introduce a thin `agc.mjs` entry that
   delegates to per-command modules. Given the binary is named `agc` in `package.json#bin`,
   a rename is not required — just an in-place extension is acceptable for MVP.

5. **No design artifacts.** This is a CLI-only feature with no visual rendering surface. No
   `design/<feature>.md` exists and none is required.

6. **Antigravity T3 confidence.** The `.antigravityrules` adapter is generated with a
   "unverified — fall back to tw_switch_role" execution profile caveat. This is intentional
   and acceptable for MVP. The adapter will not be wrong; it will be conservative.

             with loader + execution-profile sections and {{AGC_VERSION}} placeholder.
             | depends_on: none

              Update usage string to list both init and check sub-commands.
              | depends_on: T-TEMPLATES

             | depends_on: T-INIT-EXTEND

             present) and agc check (stale exit 1, current exit 0, no-adapters exit 0).

             | depends_on: T-AGC-CHECK
