- **Scope decision gate<!-- origin:start --> (v3.30.0)<!-- origin:end -->**: a transition INTO build ‚Äî `(pm, In_Progress) ‚Üí (architect, In_Progress)` or `(pm, In_Progress) ‚Üí (sr-engineer, In_Progress)` ‚Äî is blocked with `SCOPE_DECISION_REQUIRED` when `design/<active_feature>.md` is armed (`## Mode` ‚âÝ `no-design`) but no scope decision is recorded. The gate clears when EITHER `.current/feature-split.md` exists (multi-feature split recorded) OR handoff `scope_decision: single-feature` is set (single-feature attestation). The predecessor is pinned to `pm:In_Progress`, so re-entry/resume (architect‚Üísr-engineer, sr-engineer self-loop) is never re-blocked. Independent of the visual evidence gate (different edge, different artifacts; shares only the arm signal). Non-design workspaces (no design file or `## Mode` = `no-design`) pass through silently. Closes the routing-chain half of the scope-creep finding (see `content/constitution-rationale.md`) ‚Äî it does NOT stop in-context / lite-mode work that emits no transition.
- **Cut-Approval Gate<!-- origin:start --> (v3.46.0)<!-- origin:end -->**: the same build-entry edge ‚Äî `(pm, In_Progress) ‚Üí (architect, In_Progress)` or `(pm, In_Progress) ‚Üí (sr-engineer, In_Progress)` ‚Äî is blocked with `CUT_APPROVAL_REQUIRED` until `cut_approved: true` is set on the handoff. Unconditional (NOT design-arm-gated): a human must approve the ticket cut before ANY build role receives the handoff, visual feature or not. File-mode only: `cut_approved` lives in the handoff YAML frontmatter, so in SQLite/HTTP mode the gate is skipped. The predecessor is pinned to `pm:In_Progress`, so resume/re-entry (architect‚Üísr-engineer, the sr-engineer self-loop) is never re-blocked. Clearing artifact: `cut_approved: true` on the `pm:In_Progress` state write, set ONLY after the human approved the inline cut in chat. Feature-scoped re-arm: a fresh PM `In_Progress` entry, a QA-FAIL bounce back to PM, or any `active_feature` change re-arms the gate ‚Äî the cut must be re-presented and re-approved. Sanctioned writer (coordinator-attested approval): cut_approved may be set ONLY by the context that directly witnessed the human's chat-turn approval in its OWN conversation ‚Äî never from another agent's summary or relayed claim of approval. In same-context dispatch (lite, full non-subagent, or the `tw_switch_role` fallback) that witness is the PM itself; under Task-subagent dispatch ‚Äî where the PM subagent ends its turn after presenting the draft ‚Äî it is the coordinator, attesting via `tw_update_state(agent_id="pm", cut_approved: true, ...)` on the PM's still-current state tuple. The server cannot verify same-context witnessing; this is an honest, attestation-based trust boundary, not a cryptographic guarantee. Role-specific actions live in the skills (PM inline-present rules: `skill-pm.md` Gate Summary; coordinator writer obligation: `skill-coordinator.md` stop-condition 6) and MUST NOT restate this mechanism.
- **Lease-Override<!-- origin:start --> (v3.80.0)<!-- origin:end -->**: a write that would be rejected `FEATURE_LEASE_HELD` (incumbent feature non-terminal and fresh ‚Äî see `gates/feature-lease.ts`'s E1/E1A/E13 header lineage for the predicate; not restated here) may carry `lease_override: true` to bypass the lease for THAT write only, after a human present in the workspace attests the incumbent lease is dead (shipped or abandoned). Sanctioned writer (coordinator-attested, the Cut-Approval Gate trust rule): `lease_override` may be set ONLY by the context that directly witnessed the human's chat-turn attestation in its OWN conversation ‚Äî never from another agent's summary or relayed claim. In same-context dispatch that witness is the acting role itself; under Task-subagent dispatch it is the coordinator, attesting on the stranded role's still-current state tuple. Differences from cut-approval: (i) usable on **any edge** where `FEATURE_LEASE_HELD` can fire, not pinned to build-entry; (ii) a **stricter audit requirement** ‚Äî the write MUST carry `pending_notes[0]` matching `/^lease-override:/` with a human-readable reason, or it is rejected `LEASE_OVERRIDE_AUDIT_MISSING` (an unaudited bypass is never silently accepted); (iii) **transient, write-scoped** ‚Äî never persisted, never carried forward; a later write omitting it is evaluated by the normal lease predicate. File-mode only: in SQLite/HTTP mode the field is ignored and the plain lease rejection stands.
- **Bookkeeping-Write<!-- origin:start --> (v3.80.0)<!-- origin:end -->**: a same-feature write that records no forward progress (a PM crash/failure record, an administrative note) may carry `bookkeeping_write: true` to attest it is non-substantive: the server then PRESERVES the existing on-disk `last_updated` verbatim instead of stamping fresh `now()`, so the incumbent feature's lease age keeps measuring the last REAL write and a genuinely idle workspace is not re-armed for the next feature. Attested, not server-verified for truthfulness (the `scope_decision_why` trust class); server-verified for shape: valid ONLY when the write's `active_feature` equals the existing on-disk `active_feature` ‚Äî a differing-feature combination is rejected `BOOKKEEPING_WRITE_INVALID_FEATURE_CHANGE`, never silently accepted or downgraded (suppressing a NEW feature's own freshness stamp would make its lease look pre-aged ‚Äî the clobber race the lease exists to close). The `readHandoffState` schema-migration heal-write is this mechanism's **hard-wired unconditional equivalent** (server-internal, no attestation): a read-triggered heal is never evidence the feature is alive and cannot extend its lease. Transient, write-scoped; file-mode only (SQLite/HTTP timestamp behavior unchanged).
- **Amend-Resume Edge<!-- origin:start --> (v3.46.0)<!-- origin:end -->**: when PM re-enters `pm:In_Progress` mid-chain to amend a spec-only issue flagged by a downstream role, two additional guarded edges ‚Äî `pm:In_Progress ‚Üí code-reviewer:In_Progress` and `pm:In_Progress ‚Üí qa-engineer:In_Progress` ‚Äî allow PM to hand back directly to the role that was stranded, instead of a manufactured detour through sr-engineer. **Mechanism:** the write that crosses the edge must set the first-class `resume_of` handoff field to `code-reviewer` or `qa-engineer` (schema v7, c9-protocol-fields; enum-validated at the tool boundary; trust class of `scope_decision_why`: attested, not server-verified ‚Äî the server checks only that the field names the exact target role; whether that role was genuinely stranded is the writer's honest attestation). Scoped to exactly the stranded role ‚Äî the field only opens an edge to the role it names, and the static routing table still forbids `pm‚Üí{code-reviewer, qa-engineer}` absent the field, so this can never skip code-reviewer or qa-engineer on a normal forward flow. A legacy `resume_of: <role>` line inside `pending_notes` is inert ‚Äî not honored, not rejected. **It does not interact with the Scope Decision Gate or the Cut-Approval Gate**, which fire only on `pm:In_Progress ‚Üí {architect,sr-engineer}:In_Progress`; these edges target `code-reviewer`/`qa-engineer`, so neither gate arms.
- Code-reviewer approval is signalled via `(code-reviewer, In_Progress) ‚Üí (qa-engineer, In_Progress)` handoff with the first-class `review_verdict` field set to `APPROVED` and a `review_reports/review_<task-id>.md` evidence file; a rejection sets `review_verdict=CHANGES_REQUESTED` on the `(code-reviewer, FAIL)` write. The server enforces verdict‚ü∫status consistency (`REVIEW_VERDICT_STATUS_MISMATCH`). Code-reviewer cannot use `status=PASS`; that remains qa-engineer-exclusive.
- After the `review_round` cap of code-reviewer FAILs (Round 4 of `review_round`), only `(pm, In_Progress)` is accepted ‚Äî symmetric to the `qa_round` circuit breaker.
